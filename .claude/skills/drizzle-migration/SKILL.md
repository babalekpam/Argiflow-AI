# Drizzle Migration Skill

Use when `db:push` is not safe (production data, risky schema changes).

## When to Use This vs `db:push`

| Situation | Use |
|-----------|-----|
| Development, no real data at stake | `npm run db:push` |
| Adding a nullable column | `npm run db:push` |
| Production database | Migration files |
| Renaming a column | Migration files |
| Dropping a column | Migration files |
| Adding NOT NULL without default | Migration files |
| Adding an index | Migration files |

## Workflow

### 1. Understand the Change

Read the current schema:
```bash
grep -n "tableName" shared/schema.ts
```

Determine:
- What columns are being added/removed/changed?
- Is there existing data to preserve?
- Is this running on a live database with users?

### 2. Set Up Drizzle Migrations (if not already configured)

Check `drizzle.config.ts`:
```ts
import { defineConfig } from "drizzle-kit";

export default defineConfig({
  schema: "./shared/schema.ts",
  out: "./drizzle",           // migration files go here
  dialect: "postgresql",
  dbCredentials: {
    url: process.env.DATABASE_URL!,
  },
});
```

### 3. Make the Schema Change

Edit the schema file (`shared/schema.ts` or relevant file) with the desired changes. Do NOT push yet.

**Rename example:**
```ts
// Change: oldColumn → newColumn
// In pgTable():
newColumn: text("new_column"),   // add new
// Keep old column temporarily for data migration
```

**Add index example:**
```ts
import { index } from "drizzle-orm/pg-core";

export const leads = pgTable("leads", {
  userId: varchar("user_id").notNull(),
  email: text("email").notNull(),
  // ...
}, (table) => ({
  userIdIdx: index("leads_user_id_idx").on(table.userId),
  emailIdx: index("leads_email_idx").on(table.email),
}));
```

### 4. Generate the Migration

```bash
npx drizzle-kit generate
```

This creates a new file in `./drizzle/` like `0001_xxx.sql`. Read it carefully:
```bash
cat drizzle/0001_xxx.sql
```

Verify the SQL does exactly what you intended. If it drops data unexpectedly, edit the SQL file directly.

### 5. Add Data Migration (if renaming/transforming)

For column renames, the generated SQL drops+adds. Add a data migration step manually in the SQL file:
```sql
-- Generated by drizzle-kit:
ALTER TABLE leads ADD COLUMN new_column TEXT;

-- Add data migration BEFORE the drop:
UPDATE leads SET new_column = old_column;

ALTER TABLE leads DROP COLUMN old_column;
```

### 6. Apply the Migration

**Development:**
```bash
npx drizzle-kit migrate
```

**Production (via script):**
```bash
# Run migration then verify
npx drizzle-kit migrate && npm run check
```

### 7. Update Application Code

After migration, update all code that referenced the old column name:
```bash
grep -rn "oldColumn\|old_column" server/ client/ shared/ --include="*.ts" --include="*.tsx"
```

### 8. Verify

```bash
npm run check   # TypeScript clean
# Test the affected routes manually
```

### 9. Commit Migration Files

Commit the schema change AND the migration file together:
```bash
git add shared/schema.ts drizzle/0001_xxx.sql
git commit -m "db: rename old_column to new_column in leads table"
```

## Common Mistakes

- Running `db:push` after already having migration files — this can conflict. Pick one approach per project.
- Not reading the generated SQL — drizzle-kit can sometimes generate destructive statements
- Forgetting to commit migration files — teammates' DBs won't stay in sync
- Applying a migration that renames a column without first copying data → data loss
